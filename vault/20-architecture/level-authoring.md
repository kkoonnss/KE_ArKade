# Level Authoring (design as play)

Level design is part of the fun, not a chore. **Awkward procedural results are a
feature** — they make level-building playful and give each venue character. The
authoring experience must be intuitive, simple, and fast. Owner lane: **Codex**
(authoring logic + a lightweight GUI for MVP; folds into the hub as a "Design"
screen in Phase 2).

## Two MVP authoring modes (same output: a palette-v1 `semantic_map.png`)

**1. Slider mode — derive zones from a photo.**
Load a photo reference (a wall, an object, a painted surface). Live sliders drive
the compiler in real time: threshold, posterize, hue-bands, edge-detect. The
operator nudges sliders until the auto-derived semantic zones look good. Great
for "trace the real objects in the space" with zero painting.

**2. Paint mode — paint over the photo.**
Show the photo as a reference layer with an **opacity slider**. The operator
paints semantic colors (palette v1 `authoring_color`) directly over it — around
objects, along edges, wherever. **Toggle the reference off for play.** Great for
deliberate, hand-authored intent.

Both modes can mix: derive with sliders, then touch up by hand.

## Visibility during play (per-game)

The photo reference is an **authoring aid — off during play by default.**
Cartridges may expose their own visibility options (show/hide zone tints, the
grid, debug overlays). A game decides what the player sees; the authoring tool
only decides what the *designer* sees while building.

## Output contract

Authoring writes exactly what the compiler writes: a clean
`content/scenes/<scene>/levels/<level>/semantic_map.png` (palette v1) plus the
`level.yaml`. Downstream derived layers (navgraph, container/well, edges) are
regenerated by the compiler — authoring never hand-edits `derived/**`.

## Game-fit adjustments

The canonical authored image stays singular: `semantic_map.png` is the shared
truth of the room. Game compatibility tweaks are procedural settings loaded by a
cartridge at startup. Each cartridge should keep one small settings registry,
keyed by level identity, instead of saving extra maps or scattering adjustment
files through level folders.

If a cartridge needs a game-specific interpretation image, it should generate it
in memory on start from `semantic_map.png`, the shared `derived/**` files, and
its own procedural knobs. Do not save extra per-game level maps. The durable
state is the neutral semantic map plus the game's settings.

Recommended cartridge settings shape:

```json
{
  "schema": "cartridge_level_adjustments",
  "version": "1.0.0",
  "cartridge_id": "on_track",
  "levels": {
    "scene_demo_wall/demo_level": {
      "track_friction": 4.5,
      "top_speed": 560.0
    }
  }
}
```

The default storage path should be `user://level_adjustments.json` inside that
cartridge. The level key should be `<scene_id>/<level_id>` so the same game can
remember custom tuning for any authored map without duplicating the map.

The Tab menu should use a shared shell/template for consistency, but the controls
inside it should be game-specific. A racing game might expose grip, speed,
checkpoint spacing, and wall forgiveness; a platform game might expose jump
height, platform snap, climb tolerance, and hazard leniency. Some controls can
stay as designer-facing sliders, while others may be refined into fixed baked
defaults or constrained ranges once the game feels right.

This gives each cartridge room to become its own instrument while still using
the same persistence pattern.

Good shared candidates can be offered broadly when they map cleanly to many
games:

- `grid_scale` or `cell_px_override` for games that quantize the semantic map.
- `spawn_padding` for extra distance from solids/hazards after nearest-walkable placement.
- `hazard_leniency` for forgiving collision against authored hazard zones.
- `wall_forgiveness` for how aggressively solids block movement.
- `pickup_density` for games that seed collectibles from the map.
- `debug_overlay` for showing interpreted grid/path/zone data during tuning.

These are optional per cartridge. A game should only expose a shared knob when
it meaningfully affects that game's mechanics and has a safe bounded range.

Recommended per-game startup order:

1. Load `semantic_map.png` and shared `derived/**` layers.
2. Load neutral authoring metadata from `derived/authoring_profile.json` if it exists.
3. Load optional per-level procedural knobs from the cartridge's `user://level_adjustments.json`.
4. Generate any game-fit masks in memory.
5. Run spawn safety and compatibility adaptation before play begins.

## MVP scope

- Slider mode + paint mode + reference-opacity toggle, producing a valid map.
- Standalone lightweight GUI (Python) is fine for MVP; in-hub authoring is Phase 2.
- Target the **wall** MVP first (vertical photo refs, easy projector testing).

## Why this matters

It turns the "we don't want to spend forever on site" problem into a 2-minute,
genuinely fun ritual: load a photo or paint a few zones, see the games populate
the space, accept the happy accidents. That ritual is part of the product.
